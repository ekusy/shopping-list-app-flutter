# 04. 通知機能（ドラフト）

グループ内の出来事（アイテム追加・「買います」宣言・購入完了など）をメンバーに
プッシュ通知する機能。AI 提案のプッシュ配信（`../AI提案機能/03-提案の表示と通知設計.md`）と
**FCM 基盤を共有**する。

## 1. 現状

- `UserDoc` に `notificationsEnabled` / `fcmToken` が定義済みだが、
  `firebase_messaging` パッケージ未導入・トークン取得フローなし・送信側なし
  （REMAINING_TASKS にも「プッシュ通知の実送信は未対応」と明記）。
- 通知 ON/OFF の設定 UI はあるが、実体としては Firestore のフラグを変えるだけ。

## 2. 基盤と個別通知を分離する

AI 提案 Phase 2 と本機能のどちらが先になっても二重実装しないよう、2 層に分ける。

### 2.1 通知基盤（共通・一度だけ作る）

`../AI提案機能/03-提案の表示と通知設計.md` §2.2〜2.3 の内容と同一:

- `firebase_messaging` 導入、許可リクエスト（通知 ON 操作時）、
  トークン取得・`onTokenRefresh`・ログアウト/OFF 時のクリア
- プラットフォーム別設定（Android: チャンネル + POST_NOTIFICATIONS /
  iOS: APNs / Web: VAPID + SW 共存検証）
- Functions 側の送信ユーティリティ: 「グループの memberIds から
  `notificationsEnabled && fcmToken != null` のトークンを集めて
  `sendEachForMulticast`、無効トークンはクリア」を共通関数化
- タップ時のルーティング: `data: { route: '/...' }` を go_router に流す共通ハンドラ

### 2.2 イベント通知（本機能の固有部分）

Firestore トリガー（AI 提案 Phase 0 で `functions/` とアイテムトリガーが整備済みの前提）に
送信処理を追加する。候補イベントと初期スコープ:

| イベント | トリガー | 通知先 | 初期スコープ |
|---|---|---|:---:|
| 「買います」宣言（`buyingBy` が null → 非 null） | `onDocumentUpdated` | 宣言者**以外**のメンバー | ✅ 第1弾 |
| 購入完了（status active → purchased） | 同上（Phase 0 の購入イベント検知と同一箇所） | 購入者以外のメンバー | ✅ 第1弾 |
| アイテム追加 | `onDocumentCreated` | 追加者以外のメンバー | ⏸ 様子見（頻度が高く通知疲れリスク。バッチ集約（後述）とセットで） |
| メンバー参加（招待コード参加） | groups の `onDocumentUpdated`（memberIds 増加） | 既存メンバー | ✅ 第1弾 |
| AI 週次提案 | 週次パイプライン | 全メンバー | AI Phase 2 |

設計原則:

- **自分の操作は自分に通知しない**（actor uid を除外）。
- **通知疲れを最優先で警戒する**。「買い物のたびに使うアプリ」のコア体験を
  通知ノイズで壊さない（広告を採らないのと同じ思想）。第1弾は
  「相手の行動で自分の行動が変わるイベント」（買います宣言・購入完了・メンバー参加）に絞る。

### 2.3 連続操作の集約（アイテム追加を入れる場合の必須要件)

買い物前のリスト作成では 5〜10 件連続追加が普通であり、1 件ごとの通知は許容されない。
入れるなら「最後の追加から N 分（例: 10 分）イベントが途切れたらまとめて 1 通
（『◯◯さんがアイテムを5件追加しました』）」というデバウンスが必要。
実装は Cloud Tasks の遅延実行 + 直近イベント数の集計で可能だが複雑なので、
**第1弾スコープから外す判断の根拠**でもある。

## 3. 通知設定の粒度

現状は `notificationsEnabled` の全体 ON/OFF のみ。段階導入:

- **第1弾**: 全体 ON/OFF のまま（既存フィールドをそのまま尊重）。
- **第2弾**: 種類別設定 `users/{uid}.notificationPrefs: { volunteer: bool, purchased: bool, memberJoined: bool, aiSuggestions: bool }`。
  未設定キーは true 扱い（後方互換）。設定画面にトグルを追加。

## 4. 実装順の選択肢

| 案 | 順序 | 評価 |
|---|---|---|
| A | AI 提案 Phase 2 を先行（基盤を AI 側で作る）→ 本機能はイベント送信の追加のみ | AI ロードマップどおり。週1通なので通知基盤の初物としてはリスクが低い |
| B（推奨） | 本機能の第1弾を先行（基盤 + 買います/購入/参加通知）→ AI Phase 2 は送信 1 箇所の追加 | **通知の体験価値はこちらの方が直接的**（リアルタイム共有という既存のコア価値を強化）。基盤の検証機会も毎日ある |

→ 要オーナー判断。どちらでも基盤(§2.1)は共通。

## 5. 論点

| # | 論点 | 推奨 |
|---|---|---|
| 1 | 第1弾のイベント範囲 | 買います宣言・購入完了・メンバー参加の 3 つ（§2.2） |
| 2 | アイテム追加通知 | デバウンス集約が用意できるまで見送り |
| 3 | プラットフォーム順 | Android → iOS → Web（AI 提案ドラフトと同方針。Web は SW 共存検証がブロッカー候補） |
| 4 | 複数端末対応 | 初期は単一 `fcmToken`（最後の端末勝ち）を許容。`fcmTokens` サブコレクション化は需要が出てから（AI 提案ドラフト L2 と同じ） |
| 5 | 通知言語 | 受信者の言語が不明（ロケール未永続化）。当面日本語固定とし、ロケール保存の設計（AI 提案 Q4）と同時に解決 |
