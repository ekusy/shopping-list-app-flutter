# 02. 買った物の履歴確認（ドラフト）

「先週何を買ったっけ」「あれを最後に買ったのはいつ」に答える履歴閲覧機能。
サービス展開計画の Pro 機能「履歴・統計ダッシュボード」の土台でもある。

## 1. 現状

- 購入済みアイテムは一括削除で**物理削除**され、削除後は一切参照できない。
- ただし **AI 提案機能 Phase 0**（`../AI提案機能/01-履歴データ設計.md`）で
  `groups/{groupId}/itemHistory`（生イベント・180日）と
  `groups/{groupId}/purchaseHistorySummaries`（商品別累計・永続）が導入される計画。

## 2. 設計方針: AI 提案 Phase 0 のデータをそのまま使う

**本機能のために新しいデータ収集は作らない。** Phase 0 の 2 コレクションが
そのまま履歴確認のデータ源になる:

| 画面要素 | データ源 |
|---|---|
| 時系列の購入履歴（「6/8 牛乳・卵を購入」） | `itemHistory`（`type == 'purchased'`、`occurredAt` 降順） |
| 商品別の「最後に買ったのはいつ・何回買った」 | `purchaseHistorySummaries` |
| 統計（週あたり購入数の推移、よく買う商品ランキング） | 両方から導出 |

Security Rules も Phase 0 で「グループメンバーは読み取り可・書き込み不可」が
設定済みのため、**クライアントの画面実装だけで成立する**。

> したがって実装順は「AI 提案 Phase 0 → 本機能」が強く推奨。
> 本機能を先行させる場合も、スキーマは Phase 0 のものを使うこと（独自スキーマ禁止）。

## 3. 設計スケッチ

### 3.1 画面構成（段階導入）

**Step 1: 履歴タイムライン（Free でも見せる範囲の検討対象）**

- 新規 route `/history`（deferred import）。
- 週ごとにグルーピングした購入イベント一覧。`itemHistory` を `occurredAt` 範囲 +
  `type == 'purchased'` でページング取得（`limit` + カーソル。リアルタイム購読は不要、
  `get` ベースで十分）。
- 行アクション: 「リストに再追加」（既存 `addItem` 再利用）/
  「よく買う物に追加」（[01](./01-よく買う物リスト.md) 実装後）。

**Step 2: 統計ダッシュボード（Pro 専用）**

- 商品別ランキング（`purchaseCount` 上位）・平均購入サイクル・週次購入数の推移グラフ。
- サマリーは 1 商品 1 ドキュメントなので全件読みでも軽量。グラフ描画はまず
  自前描画 or 軽量パッケージで小さく始める（チャートライブラリ選定は実装時）。

### 3.2 必要な複合インデックス

- `itemHistory`: `(type ASC, occurredAt DESC)` — タイムラインのクエリ用。
  Phase 0 で予定している `(occurredAt DESC)` に追加する。

### 3.3 表示上の注意

- 生イベントは **180 日（TTL）で消える**仕様のため、タイムラインは「直近6ヶ月」と明示する。
  それ以前は商品別サマリー（最終購入日・回数）でしか遡れない。この非対称性を UI 文言で
  説明する（「6ヶ月より前の履歴は商品別の集計のみ保持されます」）。
- `deleted`（未購入のまま削除）イベントをタイムラインに出すかは初期は**出さない**を推奨
  （「買った物の履歴」という機能名と一致させる。分析的な表示は Step 2 で検討）。

## 4. Free/Pro 境界（[05](./05-マネタイズ.md) と連動）

サービス展開計画では「履歴・統計ダッシュボード」は Pro 専用。選択肢:

| 案 | Free | Pro | 評価 |
|---|---|---|---|
| A | 履歴なし | タイムライン + 統計 | 計画どおりだが、履歴ゼロだと機能の存在に気づけず Pro 導線として弱い |
| B（推奨） | タイムライン直近 2 週間のみ | 全期間（6ヶ月）+ 統計 | 価値を体験させた上で「もっと遡りたい」を Pro 導線にする |

→ 要オーナー判断（[05](./05-マネタイズ.md) の境界設計と合わせて決定）。

## 5. 論点

| # | 論点 | 推奨 |
|---|---|---|
| 1 | 実装順 | AI 提案 Phase 0 の後。データ蓄積の都合からも Phase 0 は早いほどよい |
| 2 | Free/Pro 境界 | §4 の案 B |
| 3 | 「誰が買ったか」の表示 | `itemHistory.purchasedBy` を保持済みのため表示可能。メンバー名解決は既存 `groupMemberNamesProvider` を再利用 |
| 4 | 生履歴の保持期間延長（Pro 特典として 1 年等） | TTL の `expiresAt` 計算をプラン別にすれば実現可能だが、初期は一律 180 日でシンプルに |
