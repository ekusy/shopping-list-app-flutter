# 05. マネタイズ（ドラフト）

収益モデル自体は `docs/初期検討/サービス展開計画.md` で検討済み
（広告なし / MVP は応援型 / Phase 2 でフリーミアム、決定 D3）。
本書はそれを**実装に落とすための構想**であり、戦略の再検討はしない。

## 1. 既定の戦略（サービス展開計画の要点）

| 段階 | 内容 |
|---|---|
| MVP | 投げ銭（Consumable IAP ¥120/¥370/¥610）+ 早期サポーター（Subscription ¥200/月、バッジのみ）。**機能制限なし** |
| Phase 2 | フリーミアム。Pro 個人 ¥380/月・ファミリー ¥580/月（仮置き） |
| 採らない | 広告モデル（コア体験・ペルソナ・プライバシーの3理由） |

Free/Pro 境界（抜粋）: 同時参加グループ 3 / メンバー上限 3名 / よく買う物 10件 /
写真総枚数 15枚 / 履歴・統計は Pro。

## 2. 現状のコードベース

- `Group.plan: String?` フィールドが存在（値の体系・書き込み箇所は実装時に要確認。
  おそらく未運用）。
- タグ上限「無料 5 / 有料 50」がドメインルールとして存在 — プラン分岐の先行事例。
  この判定ロジックの所在を確認し、§4 の共通機構に統合する。
- IAP・課金関連のパッケージは未導入。

## 3. 課金基盤の選択（最重要の技術判断）

| 案 | 内容 | 評価 |
|---|---|---|
| A | `in_app_purchase`（公式プラグイン）+ 自前のレシート検証 Functions | 手数料はストアのみ。**レシート検証・購読状態の同期・復元・猶予期間の処理を全て自前実装**。工数とバグリスクが大きい |
| B（推奨） | RevenueCat（`purchases_flutter`） | レシート検証・購読ライフサイクル・Webhook を肩代わり。無料枠（月次トラッキング収益 $2.5K まで）内なら追加コストなし。Webhook → Functions で Firestore に反映する経路が素直 |

> **Web の課金は当面対象外**とする。ストア IAP は Web で使えず、Stripe 等の導入は
> 規約・実装の両面で別プロジェクト級。Web ユーザーには「モバイルで購入すると
> アカウントに反映」とする（購読状態はサーバー側で持つため自然に実現可能）。

## 4. プラン状態の持ち方と強制

### 4.1 データ設計

- **正本はサーバー側**: RevenueCat Webhook（または検証 Functions）だけが書ける場所に置く。
  - 個人購読: `users/{uid}.plan`（`'free'` | `'pro'`）+ `planExpiresAt`
  - グループへの反映: 既存の `Group.plan` を「メンバーの誰かが Pro（ファミリー）なら
    グループも Pro」のように Functions が導出して書く
- クライアントは plan を**読むだけ**。Rules で `plan` / `planExpiresAt` への
  クライアント書き込みを禁止する（現状 users / groups の write ルールは本人/メンバーに
  広く許可されているため、**フィールド単位の禁止（`affectedKeys`）への変更が必要**。
  これは独立した先行タスクにできる）。

### 4.2 上限の強制（3層）

| 層 | 役割 |
|---|---|
| UI | 上限到達時に追加ボタンを無効化し、Pro 案内を表示（転換導線） |
| Security Rules | 改ざん対策の最終防衛。例: タグ作成時に既存件数を見るのは Rules では困難なので、カウンタフィールド（`tagCount` 等）を併設して判定 |
| Functions トリガー | カウンタの整合性維持（作成/削除でインクリメント）。写真枚数 `photoCount`（[03](./03-商品画像のStorage移行.md) §3）と同じ機構 |

上限値はハードコードせず `lib/core/constants/` のプラン定義（`PlanLimits`）に集約し、
タグ上限の既存実装もここへ統合する。

## 5. 実装フェーズ案

| 段階 | 内容 | 依存 |
|---|---|---|
| M0 | プラン定義の共通化（`PlanLimits`）+ タグ上限の統合 + plan フィールドの Rules 保護（実装済み・#39） | なし（今すぐ可能・後続全ての土台） |
| M1（MVP 応援型） | RevenueCat 導入 + 投げ銭（Consumable）+ サポーターバッジ（Subscription） | ストア配布が前提（Android リリース署名 / iOS 署名の残作業が先） |
| M2（Phase 2） | Pro 購読 + Free/Pro 境界の強制（グループ数・メンバー数・よく買う物・写真・履歴） | M0/M1 + 対象機能（[01](./01-よく買う物リスト.md)〜[03](./03-商品画像のStorage移行.md)）の実装 |
| M3 | ファミリープラン（グループ全員へ Pro 付与） | M2 の運用実績 |

> **M1 の前提に注意**: IAP はストア経由配布が必須。REMAINING_TASKS にある
> Android リリース署名・iOS 実機検証・ストア申請（特商法表記・プライバシーポリシー
> 含む）が先行タスクになる。マネタイズの実体は「ストアリリース作業」とほぼ一体。

## 6. AI 提案機能との関係

- AI 提案は計画時点の Free/Pro 境界表に存在しない新機能。運用コストは小さい
  （`../AI提案機能/04-コスト試算と運用.md`）ため、**Free に含めてエンゲージメント・
  差別化要素にする**のを推奨。Pro 差別化は「提案頻度・件数・履歴保持期間」で行う
  （AI 提案ドラフト Q6 と同一論点。ここで一元管理する）。
- 「履歴・統計ダッシュボード」（Pro）は [02](./02-購入履歴確認.md) §4 の境界案 B
  （Free は直近 2 週間）を推奨。

## 7. 論点

| # | 論点 | 推奨 |
|---|---|---|
| 1 | 課金基盤 | RevenueCat（§3 案 B）。脱却可能性（エンタイトルメント定義を自前抽象の裏に隠す）だけ担保 |
| 2 | Web 課金 | 当面なし。購読状態はサーバー正本なので Web でも Pro 状態は反映される |
| 3 | M0 の先行着手 | 推奨。課金と無関係に「上限ロジックの散在」を防ぐリファクタリングとして価値がある |
| 4 | 価格・境界の検証 | 計画どおりローンチ前にユーザーインタビュー（リスク R8）。本書は実装機構のみを定義し、値は `PlanLimits` で変えられる前提にする |
| 5 | AI 提案の位置づけ | Free 提供 + Pro で頻度/件数強化（§6） |
