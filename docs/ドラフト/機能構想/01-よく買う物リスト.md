# 01. よく買う物リスト（Epic-04）（ドラフト）

毎週買う定番品をテンプレートとして管理し、買い物リストへの一括/選択追加で
週末のリスト作成を数タップで終わらせる機能。

- 対応 Story: S04-01（登録）/ S04-02（全件追加）/ S04-03（選択追加）
  — `docs/初期検討/ユーザーストーリー検討.md` 参照
- ドメインモデル §5 で「Phase 2 の新エンティティ（テンプレートアイテム集）」として既出

## 1. 現状

- 該当エンティティ・画面は存在しない。
- 「購入済みアイテムの一括削除」が定番品を毎週手入力し直す運用を前提にしており、
  この手間の解消が本機能の価値。

## 2. 設計スケッチ

### 2.1 エンティティとスキーマ

**グループ単位**で持つ（ユーザー単位ではない）。リストが共有なら定番品も共有が自然で、
Free/Pro 境界（§4）もグループ単位の方が判定しやすい。

`groups/{groupId}/favoriteItems/{favoriteId}`:

| フィールド | 型 | 内容 |
|---|---|---|
| `name` | string | 商品名 |
| `tagId` | string? | 追加時に引き継ぐタグ |
| `note` | string | 追加時に引き継ぐメモ |
| `imageUrl` | string | 写真（[03](./03-商品画像のStorage移行.md) の Storage 移行後は Storage URL） |
| `order` | number | 表示順 |
| `createdAt` / `addedBy` | timestamp / string | 監査用 |

ドメインエンティティは `FavoriteItem`（`lib/domain/entities/favorite_item.dart`）として新設。
`Item` とフィールドが近いが、status / buyingBy 等の「リスト上の状態」を持たない
別概念なので流用せず分ける。

### 2.2 登録の入口（2系統）

1. **よく買う物リスト画面から直接追加**（S04-01 AC1）: 既存の `add_item_form` に近い
   フォームを再利用。
2. **買い物リストのアイテムから昇格**（S04-01 AC2）: アイテムのコンテキストメニュー等に
   「よく買う物に追加」。`Item` の name/tagId/note/imageUrl をコピーして作成。

### 2.3 リストへの追加（S04-02 / S04-03）

- 選択 UI（チェックボックス）+「全選択」で、全件追加と選択追加を 1 画面で実現する。
- **重複判定**（S04-02 AC2）: アクティブアイテムとの突合は AI 提案機能で導入する
  `nameKey` 正規化（`../AI提案機能/01-履歴データ設計.md` §4）と同じ関数を
  `lib/core/utils/` に置いて共用する。「牛乳」と「牛乳 」を別物として二重追加しない。
- 追加は既存 `ItemRepository.addItem` をバッチ的に呼ぶ（または WriteBatch 用の
  メソッドを追加）。

### 2.4 ルーティング・Rules

- 新規 route `/favorites`（deferred import、CLAUDE.md 方針どおり）。
- Security Rules は items / tags と同型（グループメンバーのみ読み書き）。

## 3. AI 提案機能との連携（あとから効く設計）

- AI 提案 Phase 0 の `purchaseHistorySummaries` は「購入回数上位の商品」をそのまま
  持っているため、**「よく買う物の候補」を自動提示**できる
  （例: 登録画面に「最近よく買っている: 牛乳・卵 … 追加しますか?」）。
  実装は任意だが、スキーマ突合のため `nameKey` の共用（§2.3)が前提。
- 逆方向: よく買う物リストの内容を週次プロンプトに含めると「定番品なのにリストにない」
  という購入忘れ判定の精度が上がる。

## 4. Free/Pro 境界（サービス展開計画より）

- Free: **10 件まで** / Pro: 無制限。
- 上限チェックはタグ上限（無料5/有料50）と同じ仕組みに乗せる。現状のタグ上限の実装箇所を
  確認し、プラン別上限の判定を共通ユーティリティ化する（[05](./05-マネタイズ.md) §4）。

## 5. 論点

| # | 論点 | 推奨 |
|---|---|---|
| 1 | グループ単位かユーザー単位か | グループ単位（上記）。「自分だけの定番」の需要が出たら Phase 3 で検討 |
| 2 | 写真の扱い | [03](./03-商品画像のStorage移行.md) の移行**後**に実装したい。Base64 のままテンプレート複製を作ると肥大データのコピーが量産される |
| 3 | 一括追加時の order | リスト先頭にまとめて挿入（買い物前の確認がしやすい）を推奨 |
| 4 | 削除済みアイテムからの昇格 | 対象外（アクティブ/購入済みのアイテムからのみ）。履歴からの昇格は [02](./02-購入履歴確認.md) の画面ができてから |
