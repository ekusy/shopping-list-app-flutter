# 03. 提案の表示と通知設計（Phase 1 / Phase 2）（ドラフト）

配信は 2 段階に分ける。FCM の土台（パッケージ・トークン管理・3 プラットフォーム設定）が
未整備のため、先にアプリ内表示のみで価値検証し、プッシュ通知は独立フェーズとする。

## 1. Phase 1: アプリ内表示

### 1.1 画面・ルーティング

- 新規 route `/suggestions`（提案画面）を追加。CLAUDE.md の方針に従い
  **`deferred as` import** で遅延ロードする（既存 screen の実装パターンを踏襲）。
- ホーム（買い物リスト画面）の AppBar 等に提案へのエントリポイントを置き、
  未読の提案がある場合はバッジ表示する。

### 1.2 状態管理（Riverpod）

- `latestSuggestionProvider`: 現在のグループの `suggestions` コレクションを
  `generatedAt` 降順 `limit(1)` で購読する `StreamProvider`。
- 既読管理: 端末ローカル（`shared_preferences` に最後に見た `weekId` を保存）で開始する。
  「メンバーごとの既読をサーバーに持つ」のは Rules が複雑になるため、必要性が出てから。

### 1.3 提案画面の構成

| セクション | 内容 | アクション |
|---|---|---|
| 購入忘れかも | `forgottenItems`（confidence 順） | 「リストに追加」ボタン → 既存の `addItem` を呼ぶ |
| 次の買い物におすすめ | `recommendedItems` | 同上 |
| フッター | 生成日時・「AI による提案です」の注記 | — |

- 「リストに追加」は**既存のアイテム追加フローを再利用するだけ**であり、
  `suggestions` ドキュメント自体への書き込みは Phase 1 では行わない
  （採用/却下のフィードバック記録は Phase 3。クライアントからの部分更新 Rules が
  必要になるため切り離す）。
- 空状態（提案なし / `status: 'empty'` / 履歴蓄積中）の文言を用意する。
- i18n: 画面の枠は easy_localization で JA/EN 対応するが、**提案本文は当面日本語固定**
  （[02](./02-週次提案パイプライン設計.md) §3.1）。

### 1.4 Security Rules（追加分ドラフト）

```text
// groups/{groupId} 配下に追加
match /suggestions/{weekId} {
  allow read: if request.auth != null
              && request.auth.uid in
                 get(/databases/$(database)/documents/groups/$(groupId))
                 .data.memberIds;
  allow write: if false;  // Functions（Admin SDK）のみ
}
```

## 2. Phase 2: FCM プッシュ通知

### 2.1 現状の土台

- `UserDoc` に `notificationsEnabled: bool` と `fcmToken: String?` が**定義済み**
  （`lib/domain/entities/user_doc.dart`）だが、`firebase_messaging` パッケージは未導入で、
  トークンを取得・保存するフローは存在しない。設定 UI のトグルが `notificationsEnabled` を
  書いているのみ（実装時に現状の挙動を再確認）。

### 2.2 クライアント側の作業

1. `firebase_messaging` を pubspec に追加。
2. 通知許可のリクエスト（`notificationsEnabled` を ON にしたタイミングで要求するのが自然。
   起動直後の許可ダイアログは避ける）。
3. トークン取得・保存: ログイン後と `onTokenRefresh` で `users/{uid}.fcmToken` を更新。
   ログアウト時・通知 OFF 時はフィールドをクリア。
4. フォアグラウンド受信時の取り扱い（アプリ内表示があるため、フォアグラウンドでは
   通知を出さずバッジ更新のみ、で開始）。

> **既知の制約（許容して開始）**: `fcmToken` が単一フィールドのため、複数端末利用時は
> 「最後にログインした端末」にしか届かない。必要になったら
> `users/{uid}/fcmTokens/{token}` サブコレクションへ拡張する（[05](./05-未解決論点と決定ログ.md)）。

### 2.3 プラットフォーム別の追加作業

| プラットフォーム | 作業 | 留意点 |
|---|---|---|
| Android | `POST_NOTIFICATIONS` ランタイム許可（API 33+）・通知チャンネル定義 | google-services.json は既存運用どおり再生成 |
| iOS | APNs 認証キーの Firebase 登録・Xcode で Push Notifications / Background Modes capability | macOS ホストでの作業（Docker 対象外）。実機検証必須 |
| Web | VAPID キー生成・`firebase-messaging-sw.js` の配置 | **offline-first PWA 戦略（`flutter_service_worker.js`）との Service Worker 共存を要検証**。firebase.json のキャッシュ制御ルール（SW は no-cache）に `firebase-messaging-sw.js` の追記が必要 |

Phase 2 は **Android から着手**し、iOS / Web は順次（リリース署名・実機検証の残作業
状況と合わせる）のを推奨する。

### 2.4 サーバー側（Functions）

週次パイプラインの保存後に追加するステップ:

1. グループの `memberIds` から各 `users/{uid}` を取得。
2. `notificationsEnabled == true && fcmToken != null` のトークンを収集。
3. `messaging.sendEachForMulticast` で送信。ペイロードは
   タイトル（例: 「今週の買い物提案」）+ 本文（購入忘れ先頭 1〜2 件の商品名）+
   `data: { route: '/suggestions' }`（タップで提案画面へ遷移）。
4. `messaging/registration-token-not-registered` エラーのトークンは
   `users/{uid}.fcmToken` をクリア（無効トークンの掃除）。
5. `status: 'empty'` の週は送信しない（通知疲れの防止）。

### 2.5 通知ポリシー

- 頻度は週 1 回固定（提案生成と 1:1）。リトライで同一週に二重送信しないよう、
  `suggestions/{weekId}` に `notifiedAt` を記録し送信前にチェックする。
- 文言はパーソナライズしすぎない（ロック画面に購買内容が出ることへの配慮。
  商品名を出すのは 1〜2 件まで）。
