# 05. 未解決論点と決定ログ（ドラフト）

設計検討で確定した方針と、オーナー判断が必要な論点を一覧化する。
判断が下りたら「状態」を更新し、確定内容を該当ドキュメントへ反映する。

## 1. 決定済み（本ドラフトの推奨方針）

| # | 論点 | 決定 | 根拠 |
|---|---|---|---|
| D1 | 履歴の捕捉方式 | 論理削除ではなく **Functions トリガーでのアーカイブ** | クライアント・既存クエリ・Rules・テストへの影響ゼロ。[01](./01-履歴データ設計.md) §1 |
| D2 | 捕捉イベント | 削除だけでなく **購入（status 遷移）も記録** | 購入サイクル分析には購入イベントが本質。削除のみだとトグル運用の定番品が欠落 |
| D3 | 集約方式 | 月次バッチではなく**トリガー内インクリメンタル集約** | バッチ Function 不要・プロンプトを初日から軽量化 |
| D4 | 生履歴の物理削除 | 専用 Function ではなく **Firestore ネイティブ TTL（180日）** | 無料・実装不要 |
| D5 | 提案の保存スコープ | `users/{uid}/suggestions` ではなく **`groups/{groupId}/suggestions`** | リストはグループ共有。提案はグループの関心事 |
| D6 | 配信の段階分け | Phase 1 アプリ内表示 → Phase 2 FCM | FCM は土台未整備で 3 プラットフォーム対応が重い。価値検証を先行 |
| D7 | スケジューラ | Cloud Scheduler + HTTP ではなく **`onSchedule`** | 構成が簡素・エンドポイント非公開 |
| D8 | 冪等性 | 提案ドキュメント ID を ISO 週番号に | 再実行 = 上書きで自然に冪等 |
| D9 | 商品名の意味的同一視 | 事前クラスタリングせず **Gemini にプロンプトで委ねる** | 機械的正規化（NFKC 等）のみ実装。費用対効果が出るまで形態素解析等は見送り |
| D10 | アイテム本体への `lastPurchasedAt` / `purchaseCount` 追加 | **追加しない**（サマリー側のみ） | アイテムは物理削除される。元構成案の「アイテムに持たせる」案は削除で消える |
| D11 | キルスイッチ | `system/config.suggestionsEnabled` | デプロイなしで停止可能に |

## 2. 要オーナー判断

| # | 論点 | 選択肢と推奨 | 状態 |
|---|---|---|---|
| Q1 | **Blaze プラン移行の承認** | 必須前提。予算アラート（例: 月 ¥1,000）とセットで移行（[04](./04-コスト試算と運用.md) §3） | 未決 |
| Q2 | 週次実行の曜日・時刻 | 推奨: 土曜 08:00 JST（週末まとめ買い前）。代案: 金曜夕方 | 未決 |
| Q3 | 生履歴の保持期間 | 推奨: 180 日。サマリーが長期傾向を持つため短くても提案品質への影響は小さい。元構成案は 6 ヶ月（≒同値）+ 2 年保存だったが、2 年の生データ保持は不要と判断 | 未決（推奨どおりなら D 化） |
| Q4 | 提案本文の言語 | 当面日本語固定（推奨）。多言語化するならユーザー/グループのロケール永続化（現状フィールドなし）の設計が先に必要 | 未決 |
| Q5 | AI 提案のオプトアウト | グループ設定に「AI 提案を無効化」を置くか。推奨: Phase 1 では置かず、要望があれば追加（履歴収集自体の停止を含むかも論点） | 未決 |
| Q6 | プラン連動（`Group.plan`） | 無料: 週次のみ / 有料: 提案件数増・頻度増 などの差別化を Phase 3 で行うか。タグ上限と同様の Free/Paid 体系に乗せるのが自然 | 未決（Phase 3 で再検討） |
| Q7 | Functions の開発環境 | docker-compose に Node サービス追加 vs flutter イメージに Node 同梱。推奨: 別サービス追加（イメージ肥大回避） | 未決 |
| Q8 | プライバシーポリシー改定 | 購買履歴の Vertex AI 送信の明記が必要（[04](./04-コスト試算と運用.md) §5）。現状ポリシー文書の所在・改定フローの確認から | 未決 |

## 3. 既知の制約（許容して進めるもの）

| # | 制約 | 影響と将来の手当て |
|---|---|---|
| L1 | 履歴のバックフィル不可 | 過去データは物理削除済み。Phase 0 デプロイ以降のみ蓄積。→ Phase 0 を最優先でデプロイし、Phase 1 開発中に履歴を貯める |
| L2 | `fcmToken` が単一フィールド | 複数端末では最後の端末にのみ通知。→ 需要が出たら `users/{uid}/fcmTokens/{token}` サブコレクションへ拡張 |
| L3 | 提案への採用/却下フィードバック未記録（Phase 1〜2） | 提案精度の改善ループは Phase 3。`suggestions` への部分更新 Rules（`affectedKeys` 制限）の設計が必要 |
| L4 | 既読管理が端末ローカル | メンバー間・端末間で既読が同期しない。→ 必要になったらサーバー側既読の設計 |
| L5 | Firestore トリガーは at-least-once | 稀な重複でサマリーの件数が僅かにずれうる。→ クールダウンで実用上吸収。厳密性が必要になったら冪等キー導入（[01](./01-履歴データ設計.md) §7） |
| L6 | Web の SW 共存が未検証 | `firebase-messaging-sw.js` と `flutter_service_worker.js`（offline-first）の共存検証は Phase 2 Web 対応時のブロッカー候補 |
